# 繰り返し入力フィールド（複数追加できる項目）仕様

作成日: 2026-06-15
ステータス: ブレインストーミング完了 / 実装承認待ち

## 1. 背景・目的

HRMOS の応募フォーム（職歴欄）のように、フォームの **回答者自身が同じ入力欄を必要な数だけ追加・削除できる** 機能を ContactHub に追加する。

参考: `https://hrmos.co/pages/kamedaseika/jobs/0000004/apply` の職歴欄

現状は各フィールドが「1つの値」を持つ前提のため、たとえば「保有資格を3つ書きたい」「希望日を複数挙げたい」といった可変個数の入力ができない。

## 2. スコープ（今回作るもの）

ブレインストーミングの結果、**シンプルな「1項目の繰り返し」** に絞る。

- 繰り返す中身は **1項目のみ**（職歴のような複数項目セットのブロック繰り返しは対象外）。
- 対応する入力種類は **テキスト（一行）** と **日付** の2種類。
- 個数ルール: **初期表示1欄／＋追加で増やせる／上限個数は管理者が設定／必須なら最低1つ入力必須**。

### 今回やらないこと（将来の拡張余地）

- 複数項目セットのブロック繰り返し（会社名＋期間＋役職…を1ブロックで追加）
- セレクト／テキストエリア等、テキスト・日付以外の型の繰り返し
- 繰り返し項目を条件分岐（logic）の参照元にすること（下記エッジケース参照）

## 3. 設計方針

### 3.1 既存フィールドへの「繰り返し許可」スイッチ方式（新フィールド型は作らない）

新しい `FieldType` を増やすのではなく、**既存の `text` / `date` フィールドに「複数追加を許可」フラグを付ける** 方式を採用する。

理由:
- Prisma の `FieldType` enum を増やさずに済み、DBマイグレーションの影響が小さい。
- `options`（JSON カラム）に設定を格納するだけで済む。
- 管理画面の操作が「既存のテキスト項目に複数追加を許可するスイッチを入れる」だけで直感的。
- 将来セレクト等に広げる場合も同じフラグを足すだけ。

### 3.2 設定の保存形式

`options`（JSON）に繰り返し設定を持たせる。

```jsonc
// text / date フィールドの options 例
{
  "repeatable": true,    // 繰り返しを有効化
  "maxItems": 10         // 上限個数（未設定なら上限なし扱い、または既定の上限）
}
```

※ 既存の `select/radio/checkbox` は `options: string[]`（選択肢配列）を使うが、`text/date` は従来 options 未使用のため、ここにオブジェクトで設定を入れても衝突しない。

### 3.3 回答値の構造

繰り返し有効時、値は **文字列の配列** で保存する（既存の `checkbox` と同じ構造）。

```jsonc
// repeatable な text/date の回答値
["値1", "値2", "値3"]
```

- 繰り返し無効時は従来どおり単一の文字列。
- 配列構造は既存の `checkbox`（string[]）で実績があり、保存・整形ロジックを流用できる。

## 5. 表示方法（「おまかせ」への推奨案）

列崩れがなく運用が楽な形を推奨する。

- **回答一覧テーブル**: 1セルにまとめ、`、`（読点）区切りで表示。→ 回答ごとに個数が違っても列数が変わらない。
- **CSV エクスポート**: 1セルにまとめ、区切り文字（例 `; ` または `／`）で連結。セル内改行は避ける。
- **回答詳細画面**: 番号付き箇条書き（1. ○○ / 2. △△）で見やすく。
- **メール通知**: 番号付き箇条書き、または各値を改行で列挙。

## 6. エッジケース・考慮点

1. **上限個数の扱い**: `maxItems` 未設定時の挙動を決める。→ 推奨: 既定上限（例 20）を設けて無制限入力を防ぐ。管理者が任意に下げられる。
2. **必須との関係**: `required: true` かつ repeatable の場合、最低1つの非空入力を必須とする。空欄のまま複数追加されたケースは送信時に空要素を除去してから検証。
3. **空欄の除去**: 送信時、空文字の要素は配列から除外して保存する（途中で削除し忘れた空欄対策）。
4. **条件分岐（logic）の参照元**: 繰り返しフィールドを条件分岐の「条件側」に使うと値が配列になり評価が複雑化するため、**今回は条件分岐の参照元としては非対応**とする（条件分岐の「対象側＝表示/非表示される側」になるのは可）。必要なら将来対応。
5. **日付の繰り返し**: 各要素を個別の日付入力欄として描画。各要素の形式バリデーションを行う。
6. **既存フォームへの影響**: repeatable 無効が既定。既存フィールドは従来どおり単一値で動作し、後方互換を保つ。
7. **埋め込みフォーム（embed-direct.js）**: 本体 FormRenderer と挙動を揃える必要がある（過去に条件分岐で本体のみ実装し埋め込みが動かなかった経緯があるため、今回は最初から両方対応する）。

## 7. 受け入れ条件（完成の定義）

- [ ] 管理画面で text/date フィールドに「複数追加を許可」「最大個数」を設定できる
- [ ] 回答画面で「＋追加」「－削除」ができ、上限で追加が止まる
- [ ] 必須時は最低1つの入力が要求される
- [ ] 確認画面・回答一覧・詳細・CSV・メールで複数値が破綻なく表示される
- [ ] 埋め込みフォーム（embed-direct.js）でも同様に動作する
- [ ] 既存フォーム（repeatable 無効）が従来どおり動く
- [ ] 型チェック（`npx tsc --noEmit`）が通る

## 8. 確定事項（2026-06-15 ユーザー確認済み）

- **個数**: 最小1・最大5個。`maxItems` の既定上限は **5**（管理者が1〜5の範囲で設定可、未設定時は5）。
- **区切り文字**: 一覧は「、」、CSV は「; 」。詳細画面・メールは番号付き箇条書き。
- **対応型**: 今回はテキスト/日付のみ（select 等への拡張は将来課題。同じ repeatable フラグを足すだけで広げられる設計を維持）。
